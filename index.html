用函数实现正则匹配中"."和"*"的功能

  • 大体思路：对字符串和模式串按位匹配，递归处理。退出递归的条件是两个字符串均为空(True)或字符串不为空且模式串为空(False)。
    问题：如果字符串为空，模式串不为空呢？
    举个栗子：
    字符串 ""，模式串"a*b*c*"，显而易见，因为此情况的存在，所以不能作为退出递归的条件。

  • 具体处理：如何按位匹配呢？可以分成两种情况，模式串有*和无*
    无*的情况很简单，只需将字符串和模式串各移动一位即可；
    有*的情况稍微有些复杂，第一种情况是*之前的字符没有匹配到，即匹配数量为0，需要将模式串移动两位，即跳过x*，字符串保持不变；第二种情况是*之前的字符匹配到了至少一次（或者是.*匹配），需要将字符串移动一位，模式串保持不变（此处匹配到一次或多次实际上是一样的）。如果匹配成功的话，最后的字符串和模式串可能会出现x, x*x这种情况，所以同时还要将模式串移动两位。

安装tornado-celery

之前为了处理耗时任务，已经在tornado中用过了concurrent.futures提供的线程池。
同时，tornado也是支持celery的，celery官方文档中给出的集成包就是tornado-celery(官方文档)
安装方法：
pip install tornado-celery
在此处有个坑，就是tornado-celery的版本是0.3.5，最后更新时间是2015年8月，所以不支持celery4.0以上版本，同时也不支持 pika (python的一个Rabbitmq客户端库) 0.10以上的版本。
所以根据需求，需要从新安装一下这两个包：
pip install celery==3.1.7
pip install pika==0.9.14

Celery学习笔记(一)

发表于 2017-10-08 | 阅读次数

Celery简介

Celery的官方文档
部分翻译版

  • 何为Celery？
    根据官方文档的说明，Celery被称作分布式任务队列，通俗的说就是异步任务调度处理。需要注意的是，Celery只处理任务，本身并没有集成消息队列，所以需要使用下面提到的broker来充当这个中间人的角色。
    Celery由三部分组成，分别是消息中间件（message broker），任务执行单元（worker）和任务执行结果存储（task result store/backend）。
    broker 是一个消息传输的中间件，可以将它理解为一个email，更具象的话可以将它看成一个消息队列。官方推荐rabbitmq。
    worker 和线(进)程池类似，负责执行Celery交付的任务，并发运行在分布式的系统节点。
    backend 即任务处理结果存储，官方推荐rabbitmq或redis。
